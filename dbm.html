Unit 1 — Introduction to DBMS

1.1 Data, Information & Database

Data
Raw, unprocessed facts and figures without context. Example: 9876543210, "Alice", 25
Information
Processed, organized data that has meaning and context. Example: "Alice's phone number is 9876543210."
Database
An organized collection of structured data stored electronically. Like a folder of related files.
DBMS
Software system used to create, manage and interact with databases. Mediates between users and data.
🧠 Memory Trick
Data → Process → Information → Store → Database → Manage → DBMS
Think: Raw wheat → Mill → Flour → Store in silo → Managed by a miller (DBMS)

1.2 What is DBMS?

A Database Management System (DBMS) is software that interacts with end users, applications, and the database itself to capture, store, and analyze data. It provides a systematic way to create, retrieve, update and manage data.

ℹ️
Key Insight
If you know one DBMS (like MySQL), you can easily transition to another (like PostgreSQL or Oracle), as they share similar concepts and functionalities.

Objectives of DBMS

  • Data Independence: Insulate users from hardware/storage changes
  • Data Integrity: Enforce accuracy and consistency of data
  • Data Security: Control access to sensitive data
  • Data Sharing: Allow multiple users concurrent access
  • Reduce Redundancy: Eliminate duplicate data
  • Backup & Recovery: Protect against data loss

1.4 Database Users

Database Administrator (DBA)
Manages the entire DBMS — creates schemas, grants permissions, handles backup/recovery, tunes performance.
Application Programmers
Write application programs that access the database via DML embedded in host languages (Java, Python, PHP).
End Users
Use applications to interact with the database. Naive users use pre-built interfaces; sophisticated users write SQL.
System Analysts / Designers
Analyze requirements, design the conceptual schema and map business rules to database structures.

1.7 DDL Commands

DDL (Data Definition Language) — defines and modifies database structure. Changes are auto-committed.

SQL — DDL: CREATE TABLE 
CREATE TABLE students (
    id        INT AUTO_INCREMENT PRIMARY KEY,
    name      VARCHAR(100) NOT NULL,
    email     VARCHAR(100) UNIQUE NOT NULL,
    gender    ENUM('Male', 'Female', 'Other'),
    dob       DATE,
    salary    INT DEFAULT 50000,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
SQL — DDL: ALTER, DROP, TRUNCATE 
-- Add a column
ALTER TABLE students ADD COLUMN city VARCHAR(100);

-- Drop a column
ALTER TABLE students DROP COLUMN city;

-- Modify column type
ALTER TABLE students MODIFY COLUMN name VARCHAR(150);

-- Rename a column (CHANGE: rename + retype)
ALTER TABLE students CHANGE COLUMN salary income BIGINT;

-- Rename a table
RENAME TABLE students TO users;

-- Remove all rows but keep structure
TRUNCATE TABLE students;

-- Remove table completely
DROP TABLE students;

Unit 2 — Relational Model, ER & EER Modeling

2.1 Relational Model Concepts

Relation
A table with rows and columns. Represents an entity. Formally, a subset of the Cartesian product of domains.
Tuple
A single row in a relation. Represents one record/instance of an entity. Formally an ordered list of values.
Attribute
A column in a relation. Represents a property of the entity. Has a name and a domain.
Domain
The set of allowed values for an attribute. Example: domain of "gender" = {Male, Female, Other}
Degree
Number of attributes (columns) in a relation.
Cardinality
Number of tuples (rows) in a relation at any point.

Relational Model Example

StudentID (PK) Name Email Gender DeptID (FK)

Degree = 5 (5 attributes), Cardinality = 3 (3 tuples)

2.2 Types of Keys

Super Key
Any attribute or set of attributes that uniquely identifies a tuple. Includes candidate keys and more.
Candidate Key
Minimal super key — no proper subset can uniquely identify tuples. A table may have multiple candidate keys.
Primary Key
One candidate key chosen by DBA to uniquely identify each row. Cannot be NULL. One per table.
Alternate Key
Candidate keys that were not chosen as the primary key. Also called secondary keys.
Foreign Key
An attribute in one table that references the primary key in another table. Establishes relationships.
Composite Key
A primary key made up of two or more attributes (columns) together uniquely identifying a row.
🧠 Memory Trick — Key Hierarchy
Super Key → Candidate Key → Primary Key → Foreign Key
Think: All athletes → Finalists → Winner → Ambassador (refers back to team)

2.4 Entity-Relationship (ER) Model

Entity
A real-world object or concept that has independent existence. Represented by a rectangle. E.g., Student, Course.
Attribute
Property of an entity. Represented by an ellipse. Types: simple, composite, multivalued, derived, key attribute.
Relationship
Association between two or more entities. Represented by a diamond. Can have attributes too.
Weak Entity
Cannot be uniquely identified by its own attributes alone. Depends on a strong entity (owner). Double rectangle.

2.5 EER Model — Extended ER

Specialization
Top-down process. A superclass entity is divided into subclasses based on some distinguishing characteristic. (IS-A relationship)
Generalization
Bottom-up process. Multiple entity types are combined into a higher-level entity type (superclass).
Inheritance
Subclasses inherit all attributes and relationships of the superclass. Can also have their own specific attributes.
Aggregation
A relationship between a relationship and an entity. Used when a relationship itself needs to participate in another relationship.

📋 ER-to-Relational Conversion Rules
▶
1
Strong Entity → Table
Each strong entity becomes a table. Key attribute becomes primary key. Simple attributes become columns.
2
Weak Entity → Table with FK
Weak entity becomes a table. Include the PK of the owner entity as FK + partial key as composite PK.
3
1:1 Relationship
Add FK of one entity into the other table (preferably with total participation side).
4
1:N Relationship
Add PK of the "1" side as FK into the "N" side table.
5
M:N Relationship
Create a new junction/bridge table with PKs of both entities as FKs. Together they form the composite PK.
6
Multivalued Attribute
Create a separate table with the entity's PK + the multivalued attribute column.

Unit 3 — Normalization

ℹ️
What is Normalization?
Normalization is the process of organizing a relational database to reduce data redundancy and improve data integrity, by decomposing tables into smaller tables while preserving the logical relationship between data.

Why Normalize? — Update Anomalies

Insertion Anomaly
Cannot insert some data without inserting other (possibly unrelated) data. E.g., Can't add a new dept without a student.
Update Anomaly
Updating one fact requires updating multiple rows. E.g., Change dept name in 100 student rows.
Deletion Anomaly
Deleting a row unintentionally deletes other important data. E.g., Deleting the last student in a dept deletes dept info.

3.1 Functional Dependencies

A Functional Dependency (FD) X → Y means: for every valid relation state, the value of X uniquely determines the value of Y.

FD TypeDefinitionExample
Trivial FDY is a subset of X: X → Y{A,B} → A
Non-trivial FDY is not a subset of XStudentID → Name
Full FDX → Y and removal of any attribute from X breaks it{SID,CID} → Grade
Partial FDX → Y but a proper subset of X also determines Y{SID,CID} → StudentName (partial: SID→Name)
Transitive FDX → Y and Y → Z implies X → Z (Y is not a key)SID→DeptID→DeptName

3.2 First Normal Form (1NF)

✅
1NF Rule
A relation is in 1NF if: All attributes contain only atomic (single, indivisible) values. No repeating groups or multivalued attributes.

✅

A relation is in 2NF if: It is in 1NF AND every non-key attribute is fully functionally dependent on the entire primary key (no partial dependencies).

3.5 Boyce-Codd Normal Form (BCNF)

⚠️
BCNF Rule (Stricter than 3NF)
A relation is in BCNF if: For every non-trivial functional dependency X → Y, X must be a superkey. Every determinant must be a candidate key.
💡
BCNF vs 3NF
3NF allows exceptions: a non-key attribute can determine a prime attribute (part of a candidate key). BCNF does not allow this. Every table in BCNF is also in 3NF, but not vice versa.

Denormalization

Denormalization is the process of intentionally introducing redundancy into a normalized database to improve read performance. Used in data warehouses and reporting systems where query speed is more important than storage efficiency.

4.1 DML — INSERT, UPDATE, DELETE

SQL — INSERT 
-- Insert a single row (specifying columns - best practice)
INSERT INTO users (name, email, gender, dob)
VALUES ('Alice', 'alice@example.com', 'Female', '1995-05-14');

-- Insert multiple rows at once
INSERT INTO users (name, email, gender, dob) VALUES
    ('Bob', 'bob@example.com', 'Male', '1990-11-23'),
    ('Carol', 'carol@example.com', 'Other', '1988-02-17'),
    ('David', 'david@example.com', 'Male', '2000-08-09');
SQL — UPDATE 
-- Update one column
UPDATE users SET name = 'Alicia' WHERE id = 1;

-- Update multiple columns
UPDATE users SET name = 'Robert', email = 'robert@example.com' WHERE id = 2;

-- Arithmetic update: increase salary by 10%
UPDATE users SET salary = salary * 1.10 WHERE salary < 60000;

-- WARNING: Without WHERE, updates ALL rows
UPDATE users SET gender = 'Other'; -- ⚠️ affects every row!
SQL — DELETE 
-- Delete one row
DELETE FROM users WHERE id = 3;

-- Delete with condition
DELETE FROM users WHERE gender = 'Other';

-- Delete all rows (keep table structure)
DELETE FROM users;        -- slow, logged, rollback-able
TRUNCATE TABLE users;    -- fast, not rollback-able

4.2 DQL — SELECT Queries

SQL — Complete SELECT Syntax 
SELECT   [DISTINCT] column_list
FROM     table_name
[WHERE    condition]
[GROUP BY column(s)]
[HAVING   group_condition]
[ORDER BY column [ASC|DESC]]
[LIMIT    n [OFFSET m]];
⚠️
SQL Query Execution Order (IMPORTANT for Exams!)
FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
SQL — WHERE Operators 
-- Comparison operators
WHERE salary > 50000
WHERE gender != 'Female'
WHERE id BETWEEN 5 AND 10

-- Pattern matching
WHERE name LIKE 'A%'       -- starts with A
WHERE name LIKE '%kumar'   -- ends with kumar
WHERE name LIKE '%li%'    -- contains li
WHERE name LIKE '_a%'     -- 2nd char is a

-- Set membership
WHERE gender IN ('Male', 'Other')
WHERE id NOT IN (1, 2, 3)

-- NULL checks (NEVER use = NULL)
WHERE dob IS NULL
WHERE dob IS NOT NULL

-- Logical operators
WHERE gender = 'Female' AND salary > 60000
WHERE gender = 'Male' OR gender = 'Other'
WHERE NOT gender = 'Female'

GROUP BY and HAVING

SQL — GROUP BY + HAVING 
-- Average salary by gender
SELECT gender, AVG(salary) AS avg_salary
FROM users
GROUP BY gender;

-- HAVING: filter after grouping (WHERE can't use aggregates)
SELECT gender, AVG(salary) AS avg_salary
FROM users
GROUP BY gender
HAVING AVG(salary) > 75000;

-- Count referrals per user (only show >1)
SELECT referred_by_id, COUNT(*) AS total
FROM users
WHERE referred_by_id IS NOT NULL
GROUP BY referred_by_id
HAVING COUNT(*) > 1;

-- ROLLUP: Subtotals and grand total
SELECT gender, COUNT(*) AS total
FROM users
GROUP BY gender WITH ROLLUP;

SQL — All JOIN Types 
-- INNER JOIN: Only rows with matches in BOTH tables
-- Output: Aarav-Mumbai, Sneha-Kolkata (Raj excluded: no address; Delhi excluded: no user)
SELECT u.name, a.city
FROM users u
INNER JOIN addresses a ON u.id = a.user_id;

-- LEFT JOIN: All from left (users) + matching from right
-- Output: Aarav-Mumbai, Sneha-Kolkata, Raj-NULL
SELECT u.name, a.city
FROM users u
LEFT JOIN addresses a ON u.id = a.user_id;

-- RIGHT JOIN: All from right (addresses) + matching from left
-- Output: Aarav-Mumbai, Sneha-Kolkata, NULL-Delhi
SELECT u.name, a.city
FROM users u
RIGHT JOIN addresses a ON u.id = a.user_id;

-- SELF JOIN: Join a table with itself (e.g. referral system)
SELECT a.name AS user_name, b.name AS referred_by
FROM users a
LEFT JOIN users b ON a.referred_by_id = b.id;

-- UNION: Combine results from two queries (removes duplicates)
SELECT name FROM users
UNION
SELECT name FROM admin_users;

-- UNION ALL: Keep duplicates
SELECT name FROM users
UNION ALL
SELECT name FROM admin_users;

4.4 SQL Functions

SQL — Aggregate Functions 
-- COUNT, SUM, AVG, MIN, MAX
SELECT COUNT(*) FROM users;
SELECT COUNT(*) FROM users WHERE gender = 'Female';
SELECT SUM(salary) AS total_payroll FROM users;
SELECT AVG(salary) AS avg_salary FROM users;
SELECT MIN(salary), MAX(salary) FROM users;

-- String Functions
SELECT LENGTH(name) FROM users;
SELECT UPPER(name), LOWER(email) FROM users;
SELECT CONCAT(name, ' <', email, '>') AS contact FROM users;

-- Date Functions
SELECT NOW();                      -- current datetime
SELECT CURDATE();                   -- current date
SELECT YEAR(dob), MONTH(dob) FROM users;
SELECT DATEDIFF(CURDATE(), dob) AS days_lived FROM users;
SELECT TIMESTAMPDIFF(YEAR, dob, CURDATE()) AS age FROM users;

-- Math Functions
SELECT ROUND(salary, 2), FLOOR(salary), CEIL(salary) FROM users;
SELECT MOD(id, 2) FROM users;

-- Conditional: IF()
SELECT name, IF(gender = 'Female', 'Yes', 'No') AS is_female FROM users;

Subqueries

SQL — Subqueries 
-- Scalar subquery: returns single value
SELECT id, name, salary
FROM users
WHERE salary > (
    SELECT AVG(salary) FROM users
);

-- Subquery with IN: returns multiple values
SELECT id, name
FROM users
WHERE referred_by_id IN (
    SELECT id FROM users WHERE salary > 75000
);

-- Subquery in SELECT (scalar)
SELECT name, salary,
    (SELECT AVG(salary) FROM users) AS avg_salary
FROM users;

4.5 Views & Indexes

SQL — Views 
-- Create a view (virtual table)

5.1 Transactions & ACID Properties

ℹ️
What is a Transaction?
A transaction is a logical unit of work that contains one or more SQL statements. It must either execute completely or not execute at all (atomicity).
Atomicity (A)
All operations in a transaction succeed or all fail (roll back). No partial execution. "All or nothing."
Consistency (C)
Transaction brings the database from one valid state to another valid state. Integrity constraints are maintained.
Isolation (I)
Concurrent transactions execute as if they were serial. Intermediate states are invisible to other transactions.
Durability (D)
Once committed, changes are permanent even if the system crashes. Written to non-volatile storage.
🧠 Memory Trick
ACID — like acid, it either reacts completely or doesn't react at all. No half-reactions!
All or nothing, Consistently valid, Isolated from others, Durable forever.

Transaction States

Transaction State Diagram
stateDiagram-v2 [*] --> Active : BEGIN / START Active --> PartiallyCommitted : Last operation executed Active --> Failed : Error/Abort PartiallyCommitted --> Committed : COMMIT (writes to disk) PartiallyCommitted --> Failed : Error during commit Failed --> Aborted : ROLLBACK complete Committed --> [*] Aborted --> [*] Aborted --> Active : Restart transaction

MySQL Transaction Commands

SQL — Transactions 
-- Disable AutoCommit (every statement is auto-committed by default)
SET autocommit = 0;

-- Make some changes
UPDATE accounts SET balance = balance - 1000 WHERE id = 1; -- debit
UPDATE accounts SET balance = balance + 1000 WHERE id = 2; -- credit

-- Save changes permanently
COMMIT;

-- OR undo if something went wrong
ROLLBACK;

-- Re-enable AutoCommit
SET autocommit = 1;

-- SAVEPOINT: partial rollback
SAVEPOINT sp1;
-- ... more operations ...
ROLLBACK TO sp1;  -- undo only to savepoint

5.2 Concurrency Control

When multiple transactions execute concurrently, problems can arise:

ProblemDescriptionExample
Dirty ReadReading uncommitted data from another transactionT1 reads T2's update before T2 commits; T2 rolls back
Lost UpdateTwo transactions update same data; one update is lostT1 and T2 both read X=100; T1 sets X=150, T2 sets X=120 — T1's update lost
Non-repeatable ReadRead same row twice, get different valuesT1 reads salary=50K; T2 updates to 60K; T1 reads again: 60K
Phantom ReadSame query returns different rows (rows added/deleted)T1 counts 10 rows; T2 inserts 5; T1 counts again: 15

Locking Protocols

Shared Lock (S-Lock)
Read lock. Multiple transactions can hold shared locks on same data simultaneously.
Exclusive Lock (X-Lock)
Write lock. Only one transaction can hold an exclusive lock. No other lock (shared or exclusive) allowed.
Two-Phase Locking (2PL)
Growing phase: acquire locks. Shrinking phase: release locks. Never acquire new locks after releasing one. Guarantees serializability.
Deadlock
Two transactions each wait for the other to release a lock. Resolved by: detection + rollback, prevention, or timeout.

ℹ️
What is Aggregation?
Like an assembly line: data passes through multiple stages, each stage transforms it and passes to the next. Similar to GROUP BY + aggregate functions in SQL.

MongoDB — Aggregation Pipeline Examples 
// Sample sales collection used below

// 1. $match — Filter (like WHERE in SQL)
db.sales.aggregate([
    { $match: { category: "Fruit" } }
])

// 2. $project — Select/reshape fields (like SELECT in SQL)
db.sales.aggregate([
    { $project: { _id: 0, item: 1, price: 1 } }
])

// 3. $group — Group and calculate (like GROUP BY)
db.sales.aggregate([{
    $group: {
        _id: "$category",
        totalSales: { $sum: { $multiply: ["$price", "$quantity"] } },
        avgPrice:   { $avg: "$price" },
        count:      { $sum: 1 }
    }
}])

// 4. $sort — Sort (like ORDER BY)
db.sales.aggregate([
    { $group: { _id: "$category", total: { $sum: "$price" } } },
    { $sort: { total: -1 } }  // -1 = descending
])

// 5. Complete pipeline: match + group + sort + limit
db.sales.aggregate([
    { $match: { category: "Fruit" } },
    { $group: { _id: "$category", totalFruitSales: { $sum: { $multiply: ["$price", "$quantity"] } } } },
    { $sort: { totalFruitSales: -1 } },
    { $limit: 5 }
])

// 6. $lookup — JOIN orders with products
db.orders.aggregate([{
    $lookup: {
        from:         "products",
        localField:   "products.name",
        foreignField: "name",
        as:           "productDetails"
    }
}])

// 7. Total revenue from all orders
db.orders.aggregate([
    { $group: { _id: null, totalRevenue: { $sum: "$total" } } }
])

Viva Questions & Model Answers

💡 Common Viva Questions
▶
Q. What is the difference between Primary Key and Unique Key?
A: Both enforce uniqueness. PK cannot be NULL, only one per table. UNIQUE allows NULL (one), can have multiple per table.
Q. What is the difference between DELETE and TRUNCATE?
A: DELETE removes rows one at a time, is logged, can have WHERE, and is rollback-able. TRUNCATE removes all rows at once, faster, resets AUTO_INCREMENT, usually not rollback-able.
Q. Can a table have multiple candidate keys?
A: Yes! A table can have multiple candidate keys. The DBA chooses one as the Primary Key; the rest become Alternate Keys.
Q. What is a NULL value? Can PK be NULL?
A: NULL means "unknown" or "not applicable" — it is not zero or empty string. Primary Key cannot be NULL (Entity Integrity Constraint).
Q. What is the difference between WHERE and HAVING?
A: WHERE filters rows before grouping; cannot use aggregate functions. HAVING filters after GROUP BY; can use aggregate functions.
Q. What is a View? Can you update data through a view?
A: A view is a virtual table (stored query). Yes, simple views on a single table without aggregates/DISTINCT can be updated. Complex views usually cannot.
Q. What is the difference between UNION and UNION ALL?
A: UNION removes duplicate rows; UNION ALL keeps all rows including duplicates. UNION ALL is faster since it skips deduplication.
Q. Explain ON DELETE CASCADE.
A: When a parent row is deleted, all child rows referencing it are automatically deleted. Prevents orphaned records.
Q. What is Normalization? Is it always good?
A: Normalization reduces redundancy and anomalies. But high normalization causes more JOINs, which can hurt query performance. Sometimes denormalization is used for speed.
Q. What is a stored procedure vs a function?
A: Stored Procedure: can return multiple results, uses OUT/INOUT parameters, called with CALL. Function: must return a single value, can be used in SELECT/WHERE.
Q. What is a Deadlock? Give a real-world example.
A: Two transactions each hold a lock and wait for the other to release. Example: T1 locks Account A, waits for B; T2 locks Account B, waits for A. Neither can proceed.
Q. What is _id in MongoDB?
A: _id is the primary key in MongoDB, automatically generated as an ObjectId (24-char hex string) if not provided. It is always unique within a collection.
Q. What is BSON?
A: Binary JSON — the binary-encoded format MongoDB uses internally to store documents. Extends JSON with additional types like Date, ObjectId, and Binary.
Q. What is the aggregation pipeline?
A: A multi-stage data processing pipeline in MongoDB. Each stage transforms documents and passes them to the next stage, similar to Unix pipes.

SQL — Lab 2: Foreign Keys and JOINs 
-- Parent table
CREATE TABLE departments (
    dept_id   VARCHAR(5) PRIMARY KEY,
    dept_name VARCHAR(100) NOT NULL,
    hod       VARCHAR(100)
);

INSERT INTO departments VALUES
    ('CS', 'Computer Science', 'Dr. Sharma'),
    ('IT', 'Information Technology', 'Dr. Patel'),
    ('EC', 'Electronics', 'Dr. Verma');

-- Child table with FK
CREATE TABLE employees (
    emp_id   INT AUTO_INCREMENT PRIMARY KEY,
    emp_name VARCHAR(100),
    salary   DECIMAL(10,2),
    dept_id  VARCHAR(5),
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id) ON DELETE SET NULL
);

INSERT INTO employees (emp_name, salary, dept_id) VALUES
    ('Alice', 75000, 'CS'),
    ('Bob', 65000, 'IT'),
    ('Carol', 80000, 'CS'),
    ('Dave', 60000, 'EC');

-- INNER JOIN: employees with their department names
SELECT e.emp_name, e.salary, d.dept_name, d.hod
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;

-- Department-wise average salary
SELECT d.dept_name, AVG(e.salary) AS avg_salary
FROM departments d
LEFT JOIN employees e ON d.dept_id = e.dept_id
GROUP BY d.dept_name;

-- Employees earning above average
SELECT emp_name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Create a view for high-salary employees
CREATE VIEW senior_employees AS
    SELECT emp_name, salary, dept_id
    FROM employees
    WHERE salary > 70000;

SELECT * FROM senior_employees;

-- Create index on emp_name for faster search
CREATE INDEX idx_emp_name ON employees(emp_name);

-- Stored Procedure: Insert new employee
DELIMITER $$
CREATE PROCEDURE InsertEmployee(
    IN p_name   VARCHAR(100),
    IN p_salary DECIMAL(10,2),
    IN p_dept   VARCHAR(5)
)
BEGIN
    INSERT INTO employees (emp_name, salary, dept_id)
    VALUES (p_name, p_salary, p_dept);
    SELECT 'Employee inserted successfully' AS message;
END$$
DELIMITER ;

CALL InsertEmployee('Eve Singh', 72000, 'IT');

-- Trigger: Log salary changes
CREATE TABLE salary_log (
    id         INT AUTO_INCREMENT PRIMARY KEY,
    emp_id     INT,
    old_salary DECIMAL(10,2),
    new_salary DECIMAL(10,2),
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

DELIMITER $$
CREATE TRIGGER after_salary_update
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    IF OLD.salary != NEW.salary THEN
        INSERT INTO salary_log (emp_id, old_salary, new_salary)
        VALUES (OLD.emp_id, OLD.salary, NEW.salary);
    END IF;
END$$
DELIMITER ;

MongoDB — Lab 4: Full CRUD + Aggregation 
// Switch to database
use college_db

// INSERT students
db.students.insertMany([
    { roll: 1, name: "Aarav", branch: "CS", marks: 88, city: "Mumbai" },
    { roll: 2, name: "Priya", branch: "IT", marks: 92, city: "Delhi" },
    { roll: 3, name: "Ravi", branch: "CS", marks: 76, city: "Pune" },
    { roll: 4, name: "Sneha", branch: "EC", marks: 85, city: "Mumbai" },
    { roll: 5, name: "Rohan", branch: "IT", marks: 91, city: "Chennai" }
])

// FIND all CS students sorted by marks
db.students.find({ branch: "CS" }).sort({ marks: -1 })

// FIND: marks between 80 and 95
db.students.find({ marks: { $gte: 80, $lte: 95 } }, { name: 1, marks: 1, _id: 0 })

// UPDATE: Give 5 bonus marks to all IT students
db.students.updateMany({ branch: "IT" }, { $inc: { marks: 5 } })

// DELETE: Remove students with marks below 70
db.students.deleteMany({ marks: { $lt: 70 } })

// AGGREGATION: Average marks per branch
db.students.aggregate([
    {
        $group: {
            _id: "$branch",
            avgMarks: { $avg: "$marks" },
            count:    { $sum: 1 }
        }
    },
    { $sort: { avgMarks: -1 } }
])

// AGGREGATION: Students from Mumbai with marks > 80
db.students.aggregate([
    { $match: { city: "Mumbai", marks: { $gt: 80 } } },
    { $project: { name: 1, marks: 1, _id: 0 } }
])
