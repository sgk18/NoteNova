Executive Summary

NoteNova is a multi-tenant, AI-augmented academic knowledge platform designed to unify the fragmented digital infrastructure of higher education institutions. It replaces the current patchwork of disconnected learning management systems, file-sharing tools, messaging platforms, and informal study channels with a single coherent system spanning the entire academic lifecycle — from content creation and course delivery to evaluation, feedback, and long-term institutional knowledge retention.

The platform serves four distinct actor classes simultaneously — students, faculty, administrators, and departments — each with deeply differentiated workflows, permissions, and objectives. NoteNova does not impose a singular workflow but adapts to the variability inherent in higher education: varying teaching methodologies, departmental structures, student engagement patterns, and institutional compliance requirements.

Scope Statement

NoteNova is scoped as a production-grade, institutional SaaS platform supporting deployment at institutions ranging from small colleges (<2,000 students) to large public universities (>50,000 students), with full data isolation between tenants, role-based access control at the institution level, and configurable academic workflow templates. This is not a consumer note-taking application; it is an institutional infrastructure product with strict compliance, audit, and data governance requirements.

Primary Objectives
  • Eliminate fragmentation across academic digital tools
  • Establish durable institutional knowledge structures
  • Automate repetitive academic workflow steps
  • Provide AI-assisted learning within ethical boundaries
  • Enable real-time collaboration across student-faculty boundaries
  • Surface actionable analytics for all actor classes
  • Enforce institutional data governance and compliance
Explicit Non-Goals
  • Consumer social networking features
  • Full LMS replacement in first version (bridge-first model)
  • AI-generated grade assignment without faculty review
  • External marketplace or public content sharing
  • Gamification disconnected from academic outcomes
  • Real-time video conferencing (integration only)
  • Financial aid or student records management

Problem Definition

Analytical Framework

This section presents a structural decomposition of problems across seven dimensions. For each dimension, root causes, failure modes, and downstream effects are identified. Understanding WHY these problems persist is prerequisite to designing systems that address structural — not merely symptomatic — issues.

2.1 Structural Fragmentation

Root Cause: Higher education institutions evolved their digital tooling incrementally rather than architecturally. Each department, or even individual faculty member, adopted tools that solved immediate local problems. The result is an ecosystem where student notes live in personal cloud drives, course content lives in an LMS, announcements flow through email, collaboration happens in messaging apps, and submission management exists in yet another standalone tool. No canonical data model exists across these systems.

Cause → Effect Chain: Because no unified data model exists, integrations between tools are either absent or brittle point-to-point connections maintained by individual administrators. When tools update, integrations break silently. Students and faculty experience this as unpredictable data loss, duplicate effort, and high cognitive switching costs. Institutional knowledge is not owned by the institution — it is distributed across proprietary third-party systems with inconsistent export capabilities.

Long-Term Impact: Institutional knowledge becomes inaccessible or lost at each academic year's conclusion. When a faculty member departs, their course materials, annotations, and pedagogical structures leave with their personal accounts. There is no accumulation of institutional intelligence; each semester is effectively a cold start.

Failure Pattern

Fragmented tooling produces a "gravity well" effect: the more tools an institution adopts without integration, the higher the coordination cost, causing actors to fall back to the lowest-friction option (email, messaging apps) — which has the lowest institutional value retention.

2.2 Workflow Inefficiencies

Root Cause: Academic workflows — assignment creation, distribution, submission, grading, feedback, and revision — involve multiple actors performing sequential steps with no automated handoff mechanism. Faculty manually create assignments in one system, post announcements in another, collect submissions in a third, grade in a spreadsheet, and provide feedback through email. Each transition is a manual action with potential for delay, omission, or error.

Where Time Is Lost
  • Faculty: Manually duplicating course structures each semester, reformatting materials for different platforms, re-entering grade data, chasing missing submissions
  • Students: Aggregating information from multiple sources, reformatting notes, re-discovering previously found material, tracking deadlines across unlinked platforms
  • Admins: Reconciling enrollment data, managing access control across systems, generating compliance reports from multiple disconnected data sources
Bottleneck Anatomy
  • Submission-to-Feedback lag: Average 7–14 days; no structural mechanism for partial feedback or draft review cycles
  • Course Setup: Repeated each semester without template-based automation; estimated 8–20 hours per course per faculty
  • Grade Reconciliation: Manual export/import cycles between LMS and institutional records systems introduce data integrity risks at each transfer

2.3 Knowledge Loss and Non-Persistence

Root Cause: No institutional system is designed to accumulate and structure knowledge over time. Notes created by students are stored in personal accounts. Faculty annotations on student work disappear at course end. Discussion threads in LMS forums expire or become inaccessible after enrollment periods close. The cumulative effect is that an institution with 40 years of pedagogical history has effectively no structured access to that history.

Institutional Impact: Curriculum development suffers from lack of historical data. New faculty cannot build on the work of predecessors. Students cannot access materials from prior cohorts to understand subject trajectories. Institutional research into learning outcomes has no structured data substrate to operate on.

Compounding Effect

Knowledge loss is not linear. Each semester of non-persistence multiplies the structural debt. An institution that has operated for 10 semesters with disconnected tooling has accumulated knowledge loss equivalent to the pedagogical investment of those 10 semesters — which is unrecoverable without deliberate re-documentation.

2.4 User Behavior Problems

S
Students
Prefer informal tools

Students adopt informal tools because institutional systems create friction disproportionate to perceived value. When a student can capture a thought in a messaging app in 2 seconds but must log into an LMS, navigate to the correct course, and find a submission field — the informal tool wins every time. This is not laziness; it is rational tool selection under cognitive load constraints.

F
Faculty
Avoid complexity

Faculty are subject-matter experts, not system administrators. When platforms require configuration expertise to use effectively, faculty revert to the path of least resistance: email. Academic promotion structures reward research and teaching quality, not technology engagement — so there is no institutional incentive to invest time in learning complex platforms.

A
Admins
Compliance-driven

Administrators operate under compliance pressure. Their primary concern is audit-readiness, enrollment accuracy, and regulatory reporting. Systems that prioritize user experience over compliance features are viewed as risks. Admins will not adopt tools that lack clear audit trails, role delineation, and institutional data control mechanisms.

2.5 Engagement Failure

Root Cause: Institutional academic platforms are passive repositories. They store documents and track submissions but do not create return loops. A student who submits an assignment has no reason to re-engage with the platform until the next assignment is due. There is no mechanism that rewards revisitation, surfaces relevant prior knowledge, or connects a student's current activity to previous work.

Why Users Do Not Return: Systems without active value generation become cost centers in the user's mental model. If each visit requires navigating toward a goal with no ambient value discovery, the system is experienced as friction rather than infrastructure. Academic platforms fail because they function as filing cabinets rather than learning environments.

Design Implication

Return loops must be designed into the system architecture, not bolted on as notifications. Meaningful re-engagement occurs when the system surfaces something the user needs — not when it reminds them to engage with something they would prefer to defer.

2.6 Data and Insight Gaps

Root Cause: Because academic data is distributed across systems with no unified schema, generating meaningful institutional analytics requires either expensive integration projects or manual aggregation. The result is that faculty cannot see which topics students struggle with, admins cannot predict dropout risk before it manifests in grades, and departments cannot evaluate curriculum effectiveness with data.

Missing Visibility
  • Engagement patterns within course materials
  • Comparative difficulty across topics per cohort
  • Correlation between resource usage and outcomes
  • Time-on-task per assignment type
  • Collaboration network effects on performance
Missing Predictive Capability
  • Early warning indicators for academic risk
  • Assignment difficulty calibration signals
  • Knowledge gap identification before assessment
  • Curriculum pacing recommendations
  • Peer group learning cluster identification

2.7 Scaling Limitations

Root Cause: Point solutions adopted by institutions are typically not designed for the scale variability of higher education. A system built for 500 concurrent users will degrade unpredictably during registration periods, exam seasons, or large synchronous class events. There is no institutional mechanism for capacity planning tied to the academic calendar.

Furthermore, platform rigidity prevents institutions from adapting systems to their specific pedagogical models. A law school using Socratic method instruction has fundamentally different knowledge structuring needs than an engineering department with lab-based project workflows. Generic systems serve neither well, forcing both to operate within constraints that do not match their actual teaching practice.

Product Vision

NoteNova is the connective tissue of academic knowledge — a platform where every piece of information created within an institution becomes part of a living, searchable, AI-augmented knowledge infrastructure that serves students, faculty, and administrators in a unified, durable system.

3.1 Strategic Principles

Knowledge First

Every feature is evaluated by its contribution to durable knowledge structures. Short-lived ephemeral interactions should produce structured artifacts that persist and are discoverable by future users of the same institution.

Friction Reduction

The platform must reduce the cognitive and mechanical cost of academic work for all actors. Friction that cannot be eliminated must be made transparent and understandable, with clear indication of why it exists.

Institutional Sovereignty

Institutions own their data absolutely. Data must be exportable in open formats, never used to train shared models without explicit consent, and isolated at the infrastructure level from all other tenants.

AI as Augmentation

AI capabilities accelerate and surface — but never replace — human academic judgment. All AI-generated content is marked, attributable, and overridable by human actors at every point in the workflow.

Adaptive Structure

The system must accommodate multiple pedagogical models without requiring re-architecture. Configuration, not per-tenant code customization, is the primary adaptation mechanism for institutional variation.

Composable Integration

NoteNova integrates with existing institutional systems through standards-based protocols (LTI, SAML, SCIM, REST) rather than requiring displacement of existing infrastructure investments.

3.2 Content Lifecycle Model

Each stage of the content lifecycle has explicit system support engineered into the platform architecture. Creation tools minimize overhead. Structuring is automatable via AI tagging and hierarchy suggestions. Discovery surfaces content proactively, not only on explicit search. Consumption is measurable at the granular level. Evolution tracks provenance across versions and authors. Archive and reuse makes institutional knowledge accessible to future cohorts.

Purpose
The primary content creation surface. Supports structured rich-text document authoring for students and faculty. Accommodates unstructured capture and progressive structuring without forcing users to decide on structure upfront.

Drag-and-drop block reordering. Word count and reading time. Focus mode. Split-pane (outline + content). Markdown import/export. Command palette (Cmd+K). 100+ undo history states. Document-level comments. Inline highlight with color coding. Auto-save every 30 seconds and on blur. Real-time sync indicator.

UI Behavior
Toolbar appears on text selection. Slash menu appears on empty line with "/". Block handles appear on hover (left margin). Backlink suggestions appear in floating popup with 300ms debounce. Autosave indicator shows spinner then checkmark; on failure shows warning with retry. Outline panel syncs scroll with editor. Mentions autocomplete triggers on "@" with user/page suggestions from within same tenant.
Edge Cases

  • Concurrent edits without real-time collab enabled: show conflict notice with version diff and merge option
  • Pasting HTML from external sources: sanitize and convert to block model; strip unsafe attributes
  • LaTeX rendering failure: show raw source with error indicator inline; do not break surrounding content
  • Backlink target deleted: dead links shown with [Deleted] indicator and restoration option
  • Unsaved changes on navigation: intercept with confirmation dialog showing diff summary
Failure Scenarios
Network loss during edit: queue changes locally via IndexedDB; show offline indicator; sync on reconnect with conflict detection. Save failure (server error): retain in-memory state; retry with exponential backoff; notify user after 3 consecutive failures. Corrupt block state: isolate affected block; provide recovery from last clean snapshot.

M-02 Content Structuring Knowledge Core
Purpose
Transform unstructured or loosely organized notes and documents into navigable, hierarchical, linked knowledge structures through both manual and AI-assisted organization mechanisms.
Features
Nested page hierarchy (unlimited depth; max recommended 5 levels). Drag-and-drop hierarchy reordering. Auto-generated table of contents. Content outline panel. Section folding. Breadcrumb navigation. Page nesting rules (parent/child). Bulk move operations. Multiple hierarchy views: tree, flat list, kanban board. AI-suggested hierarchy based on content analysis. Content grouping by date, tag, or topic cluster. Cross-hierarchy linking without breaking structure integrity.
Edge Cases
Circular hierarchy attempts: blocked with clear error. Moving a parent to its own descendant: prevented; UI shows hierarchy visualization. Very deep nesting (>10 levels): warn at 7 levels. AI-suggested structure conflicting with existing user structure: present as suggestion only — never auto-apply.
Variants
Course hierarchy (faculty-managed), personal workspace (student-managed), department knowledge base (admin-governed), project hierarchy (student group-managed).
M-03 Knowledge Graph Knowledge Core
Purpose
Expose the relational structure of knowledge within a user's workspace or course via an interactive graph that surfaces unexpected connections and encourages non-linear exploration of academic content.
Features
Force-directed graph rendering (WebGL/Canvas). Node types: note, course, assignment, resource, person. Edge types: explicit link, AI-inferred semantic relation, shared tag, co-authorship. Filter panel by node type, date, tag, connection degree. Graph zoom / pan / cluster. Node detail on hover. Open node in editor on click. Orphan node detection. Cluster detection (topic groups). Ego-graph view (N-degree connections from selected node). Export graph as SVG or JSON.
Edge Cases
Graphs >5,000 nodes: progressive loading with level-of-detail rendering. No connections exist: empty state with onboarding prompt. AI-inferred edges with low confidence: shown as dashed lines with confidence score tooltip. Deleted node with existing links: rendered as tombstone node with restoration option.
M-04 Version Control Knowledge Core
Purpose
Maintain a complete, navigable history of all document changes with ability to compare versions, restore previous states, and attribute changes to specific authors with full provenance tracking.
Features
Automatic version snapshots every 15 minutes of activity plus on manual save. Named versions (milestone snapshots). Visual diff viewer (word-level change highlighting). Author attribution per change. Branch and merge for collaborative documents. Restore to any version (non-destructive: creates new version). Version comparison timeline. Storage-efficient delta storage. Retention policy governed by tenant config. Submission-lock versions (immutable once submitted for grading).
Edge Cases
Version explosion for very active documents: automatic pruning of intermediate versions older than 30 days (named versions retained permanently). Restore conflict with concurrent edit: create restore-branch; user resolves merge. Diff rendering for non-text content (images, embeds): show presence/absence; binary content not diffed inline.
Failure Scenarios
Version storage failure: queue version creation; retry; if persistent, warn user with advisory to manually save externally. Version retrieval failure: show error with support reference; do not show partial diffs that may mislead the user.
M-05 Rich Media Knowledge Core
Purpose
Support embedding, uploading, and referencing non-text content within notes and course materials including images, audio, video, PDFs, and external embeds.
Features
File upload via drag-and-drop and clipboard paste. Supported types: images (PNG, JPG, GIF, SVG, WebP), audio (MP3, M4A, WAV), video (MP4, WebM), PDFs (inline render), Office files (preview only). Max file size per tenant config (default 50 MB). Image resizing within editor. Image alt-text for accessibility. Audio/video player with speed control and timestamp notes. PDF viewer with page navigation and highlight annotation. External embed support (YouTube, Vimeo, Google Docs, Figma). CDN-served assets with signed URLs. AI-generated image descriptions for accessibility compliance.
Edge Cases
Upload exceeds limit: clear error with file size shown; partial upload prevention. Unsupported format: clear error with supported types list. External embed source unavailable: show fallback frame with original URL and error state. Transcription failure: show raw media without transcript; log for retry. File deleted after reference in document: show broken reference indicator with restore or remove options.

Modules: Search & Discovery

M-06 Semantic Search Search
Purpose
Enable meaning-based retrieval of content across the user's accessible corpus, transcending keyword matching to surface conceptually relevant results even when exact terms are absent from the query.
Features
Hybrid search combining BM25 keyword and vector embedding retrieval. Per-tenant vector namespace. Search scope controls: personal, course, department, institution. Faceted filtering by content type, author, date range, tag, and course. Result ranking by relevance score plus recency signal. Result snippets with highlighted match context. Instant results (<200ms keyword; <800ms semantic). Query suggestions / autocomplete. User-scoped search history. Saved searches with change alerting. Advanced query syntax (AND, OR, NOT, field:value). Image and attachment content indexed via OCR and AI description. Search analytics surfacing common student queries to faculty.
Edge Cases
Empty result set: show related suggestions based on query semantics. Very large indexes (>10M documents): cursor-based pagination; 100 results per page max. Query terms matching restricted content: filter from results without exposing existence. Search during index rebuild: serve from stale index with warning banner. Cross-language queries: language detection plus multilingual embedding model.
Failure Scenarios
Search service unavailable: fallback to basic keyword search in local cache. Embedding generation failure for new content: queue re-indexing; content appears in keyword search only until resolved.
M-07 Tag Taxonomy Search
Purpose
Provide a multi-level, institutionally governable tagging system enabling both free-form personal tagging and structured institutional taxonomy for course, department, and accreditation classification.
Features
Three tag scopes: personal (user-only), course (faculty-defined), institutional (admin-defined). Tag hierarchy with parent/child relationships. Tag autocomplete with usage frequency indicator. Bulk tag operations. AI-suggested tags on content creation. Tag merge and rename (institutional scope requires admin). Tag analytics: frequency and co-occurrence. Color coding per scope. Tag-based content collections. Mandatory tags for submission types configurable per course.
Edge Cases
Tag name collision across scopes: disambiguate with scope prefix in display. Tag renamed: all existing references updated atomically. Tag deleted with existing content: prompt to reassign or orphan; never silently remove tag from content. Very large taxonomy (>10,000 tags): virtual scroll in tag picker with search-first UX.
M-08 Content Discovery Search
Purpose
Proactively surface relevant content to users without explicit search, creating return loops that keep the platform contextually useful between active sessions and drive organic re-engagement.
Features
Daily digest of recently updated relevant content (email plus in-app). Contextual "you might also need" panel surfaced during editing. Related content sidebar based on semantic similarity to current document. Recently accessed section with quick-return links. Persistent "continuing where you left off" state per device. Trending in course (faculty-curated or usage-derived). New faculty-published content alerts. Content from similar students in same course (opt-in, anonymized). Weekly knowledge graph growth summary.
Edge Cases
User with no activity: show course-based defaults, not personalized recommendations. Recommendations pointing to inaccessible content: filter before surfacing; never show titles of restricted content. Recommendation quality below confidence threshold: fall back to recency-based surfacing.

M-09 AI Summarizer AI
Purpose
Generate structured summaries of student notes, course materials, and reading resources at configurable lengths and detail levels, reducing review time while preserving conceptual density.
Features
One-click document summarization. Summary length presets: 3-sentence, paragraph, detailed outline. Summary format options: prose, bullet points, structured outline with headers. Custom summary prompt in advanced mode allowing user to specify focus area. Multi-document summary synthesis. Summary of student submissions for faculty review. AI-generated label clearly applied to all summaries. Summary appended as separate section or sidebar panel. Regenerate with different settings. Summary quality feedback (thumbs up/down).
Edge Cases
Very short document (<200 words): decline to summarize; offer to elaborate instead. Document with mostly non-text content: summarize text components; note non-text elements. Summarization of sensitive content flagged by content policy: decline with explanation. Rate limit reached: queue request with estimated wait time. Hallucination detection heuristic: cross-reference key facts with source; flag discrepancies with source-grounded alternatives.
Variants
Submission summary for faculty; lecture notes summary for students; reading list synthesis for research contexts; cross-course topic synthesis for department-level curriculum review.
M-10 AI Tutor AI
Purpose
Provide a contextually grounded conversational AI assistant that answers student questions about course material using retrieval-augmented generation scoped to the course knowledge base, not general internet knowledge.
Features
Course-scoped RAG chat interface. Citations displayed inline with links to exact source document section. Persistent conversation history within session with optional long-term retention. Clarification prompts for ambiguous questions. Difficulty-aware responses adjustable to beginner/intermediate/advanced levels. Faculty-defined system prompt per course defining topic boundaries. AI refusal for out-of-scope questions with redirect to appropriate resource. Faculty monitoring dashboard showing question clusters and low-confidence responses. Daily usage limits per tenant config. Conversation export to personal note.
Edge Cases
Question not answerable from course material: clearly state limitation; suggest faculty contact or syllabus-listed external resource. Harmful query (cheating facilitation): policy enforcement; log event; respond with policy explanation. Conflicting information in source documents: surface both perspectives with source attribution. Architectural Principle

All AI capabilities in NoteNova operate within a Retrieval-Augmented Generation (RAG) architecture scoped to the tenant knowledge base. The system does not use general internet retrieval for student-facing academic assistance. All AI-generated content is attributable, clearly labeled, and overridable by human actors.

17.2 Ingestion Pipeline Detail

Content Ingestion Steps
  1. Text Extraction: Raw content extracted from block model. Embedded images processed through OCR. Audio/video transcription queued asynchronously with separate job worker.
  2. Chunking: Content segmented into 512-token overlapping chunks (128-token overlap) using semantic boundary detection with sentence and paragraph awareness to avoid mid-sentence splits.
  3. Embedding Generation: Each chunk passed to embedding model (text-embedding-3-large or equivalent). Embeddings are 1,536-dimensional float32 vectors normalized for cosine similarity.
  4. Metadata Tagging: Each chunk tagged with: tenant_id, user_id, document_id, document_type, course_id, chunk_index, created_at, access_level.
  5. Vector Storage: Embeddings stored in tenant namespace in pgvector. Full-text content stored in Elasticsearch with metadata for hybrid retrieval path.
  6. Cache Invalidation: Relevant search and summary caches invalidated for affected query patterns on new content ingestion.

17.3 Retrieval Pipeline Detail

Query to Response Steps
  1. Query Embedding: User query embedded using the same model as document embeddings to ensure vector space compatibility.
  2. Hybrid Search: Cosine similarity search in vector store (top-K=20, threshold=0.70) combined with BM25 keyword search in Elasticsearch. Results merged using Reciprocal Rank Fusion (RRF).
  3. Access Filtering: Tenant and permission filter applied to all retrieval results before generation. Cross-tenant retrieval is impossible by construction of the namespace isolation model.
  4. Reranking: Cross-encoder reranking model applied to top-20 candidates; final top-5 selected for context window assembly.
  5. Context Assembly: Retrieved chunks assembled into context window with source attribution metadata preserved for citation generation.
  6. Generation: Context plus query passed to LLM with structured system prompt defining response format, citation requirement, hallucination-reduction instructions, and refusal criteria.
  7. Citation Extraction: Generated response parsed for citation markers; source references resolved and hyperlinked in final response delivered to user.

18.3 Data Architecture

PostgreSQL — Primary Relational

All structured business data: users, courses, assignments, submissions, grades, audit logs. Row-Level Security (RLS) policies enforce tenant isolation at database level. Logical replication to read replicas. Connection pooling via PgBouncer.

Key tables: tenants, users, roles, permissions, courses, modules, assignments, submissions, grades, comments, tags, events, audit_log

pgvector — Embedding Store

Embedding storage co-located with PostgreSQL for simplified operational management. HNSW index for approximate nearest neighbor search. Namespace isolation via tenant_id column in all queries. Embedding dimensions: 1,536.

Key table: document_embeddings (tenant_id, doc_id, chunk_index, embedding, metadata_json)

Elasticsearch — Full-Text Search

Full-text search index for keyword retrieval leg of hybrid search. Index-per-tenant strategy for tenants <5,000 documents; shared index with routing for smaller tenants. Autocomplete via edge ngram analyzer. Faceted aggregations for filter counts.

Redis — Cache & Pub/Sub

Session storage, API response caching, rate limit counters, real-time event pub/sub for collaboration service. Cluster mode for horizontal scaling. Key TTL aligned to data freshness requirements per resource type. No sensitive PII in cache without encryption.

S3-Compatible — Object Storage

File attachments, media uploads, exports, submission archives, AI-generated reports. Tenant-scoped path prefix. Pre-signed URLs for direct client upload bypassing API for large files. Lifecycle policies for archival and deletion aligned to tenant data retention configuration.

TimescaleDB — Analytics

Time-series event data: user activity, engagement metrics, AI usage. Hypertable partitioning by time. Continuous aggregates for dashboard query performance. Data retention: raw events 90 days; aggregates 5 years by default.

System-Level Edge Cases

System-level edge cases are scenarios that span multiple modules, require coordinated behavior from two or more services, and often expose implicit assumptions in the architecture. Every edge case below has a defined expected behavior, a system-level response specification, and a test reference so it can be regression-tested.

Scope Note

Module-level edge cases are documented within each module definition. This section covers cross-module and system-wide scenarios that require coordination across services and do not belong to any single module.

EC-01: Institutional Merger or Acquisition ›

Scenario: Two institutions that are both NoteNova tenants merge into a single legal entity and need to operate as a single NoteNova tenant.

Expected System Behavior: This is not a self-service operation. A NoteNova platform engineer executes the migration with the following steps: (1) Both tenants enter a "merge-pending" read-only state — no new content can be created during migration window; (2) User deduplication runs: match on institutional email address, then on first/last name + student ID combination. Conflicts (users with different emails on each tenant) queued for admin resolution. (3) Content migration: all courses, notes, assignments, submissions, and vault content from the secondary tenant are migrated to the primary tenant namespace with a provenance tag indicating origin tenant. (4) SSO reconfigured for the merged institutional IdP. (5) Department structure merged: admin reviews and resolves duplicate department names. (6) Both tenants' data retention and AI policies reviewed — the more restrictive policy prevails during transition. (7) Migration validated by running the full isolation test suite against the merged tenant. (8) Secondary tenant decommissioned; URL redirected to primary tenant. (9) All users notified.

Failure Handling: Rollback is available until Step 7 completes. After Step 7, migration is committed. Rollback after commitment requires restoring from pre-migration backup (RTO: 4 hours). Pre-migration backup taken automatically at migration start.

EC-02: Student Unenrolled During Active Coursework ›

Scenario: A student is administratively unenrolled from a course mid-semester — grade dispute, registration error, academic withdrawal, or disciplinary action.

Expected System Behavior: The SCIM deprovision event or admin action triggers an immediate permission revocation cascade: (1) Course access revoked within 60 seconds; all active sessions for that course context invalidated; (2) All submitted work, grades, and integrity records retained in course records — not deleted; (3) Any faculty grading of that student's work in progress is preserved and completable; (4) Faculty receives an in-app notification: "[Student Name] has been unenrolled from [Course Name] effective [timestamp]. Their submission history is preserved for records." (5) If student was part of a collaborative study group with shared documents: student removed from group membership; their contributions to shared documents preserved but attributed to "Former Member"; group leader notified; (6) Student's personal notes created during course enrollment remain in their personal workspace under a "[Course Name] — Archived" folder; (7) If re-enrolled later, all submission history, grades, and group contributions are re-associated with their account automatically on re-enrollment.

Edge within the edge: If the student has an in-progress quiz attempt when unenrollment occurs, the attempt is auto-submitted with current answers at the moment of unenrollment. Faculty notified. Time served recorded.

EC-03: AI Service Provider Outage or Migration ›

Scenario A — Unplanned outage: Primary AI model provider returns 5xx errors for >5 consecutive minutes.

System Response: LLM Router detects failure via health check (run every 30 seconds). After 3 consecutive failures: (1) Automatically switches active AI features to secondary provider if configured; (2) If no secondary: enters cached-response mode — AI Tutor returns cached responses for common queries (last-7-day cache); AI Summarizer disabled with user message; Auto-tagging disabled; Study Planner uses yesterday's cached plan; (3) If cache miss rate >80%: AI features enter graceful degradation mode — users shown "AI features temporarily unavailable; non-AI features are fully operational"; (4) Tenant Super Admin emailed with incident details and ETA if known; (5) Core platform (notes, submission, grading) continues fully functional — AI is augmentation, not a dependency of core flows.

Scenario B — Planned model migration: NoteNova is migrating from Model A to Model B. (1) Model B deployed in shadow mode: all queries answered by both models simultaneously for 72-hour evaluation period; (2) Response quality metrics compared (citation accuracy, response coherence, refusal rate) on golden dataset; (3) If Model B passes quality gates: traffic shifted 5% → 20% → 100% over 7 days; (4) Tenant admins notified of model change with summary of quality evaluation; (5) Full re-ingestion of all tenant content required (new embedding model may have different vector space); re-ingestion runs as background job for up to 48 hours; content accessible throughout but semantic search uses old embeddings until re-ingestion complete (staleness indicator shown).

EC-04: Mass Concurrent Submission Under Load ›

Scenario: A course with 800 students has a single assignment deadline. 600 students attempt to submit within 8 minutes of each other.

Pre-emptive system behavior: Academic calendar integration detects deadlines for large courses (>200 students) 30 minutes in advance. Submission service auto-scales +4 pods at T-30 minutes. Database connection pool expanded. Redis rate limiter thresholds elevated for this tenant for the 45-minute window around the deadline.

During the burst: (1) Each submission request timestamped at API gateway receipt — this is the canonical submission time, not the DB write time, ensuring fairness under queue pressure; (2) Submission records queued in durable message queue (Kafka) before DB write — guaranteed delivery; (3) S3 file uploads happen in parallel via pre-signed URLs (direct client-to-S3), not through the API, preventing API bandwidth saturation; (4) Integrity check jobs queued asynchronously; students receive confirmation receipt before integrity check completes; (5) Faculty submission inbox updates via WebSocket every 10 seconds during burst rather than real-time to prevent WebSocket overload; (6) Late submission enforcement still applied based on gateway timestamp — a submission received at the gateway at 11:59:59 is on-time even if the DB write completes at 12:00:03.

Post-burst: All queued submissions processed within 5 minutes of burst peak. Integrity checks completed within 30 minutes. Faculty notified of final submission count.

EC-05: GDPR Right to Erasure Request ›

Scenario: A student (EU data subject) submits a verified Article 17 GDPR erasure request through their institution. The institution submits it to NoteNova via the admin API.

System Response — 30-day process:

  1. Request intake (Day 0): Admin submits erasure request via API with verified identity token. System creates an erasure_request record with status: pending. All user sessions revoked immediately. User account locked (no login). Audit log entry created.
  2. Data inventory (Days 0–2): Automated scan across all data stores: PostgreSQL (user record, notes, submissions, comments, activity events, quiz attempts), pgvector (all document embeddings for this user's content), Elasticsearch (full-text index entries), S3 (uploaded files, submission archives), Redis (cached session data, recommendation cache), TimescaleDB (activity event stream). Full inventory report generated and retained for audit.
  3. Legal basis review (Days 2–7): System classifies each data category: (a) No legal basis for retention → scheduled for deletion; (b) Legitimate interest / legal obligation → retained with basis documented (academic records, grade records, integrity incident records may have statutory retention requirements depending on jurisdiction). Institution's legal team reviews the retained-data list.
  4. Staged deletion (Days 7–14): Category (a) data deleted: personal notes (permanently deleted from DB + S3), cached data purged, embeddings deleted from vector store, activity events deleted from TimescaleDB, comment author IDs replaced with "Deleted User." Category (b) data anonymized where possible (name and email replaced with pseudonyms; student ID retained for record linkage).
  5. Certificate generation (Day 14–15): Deletion certificate generated as PDF: lists all deleted data categories with deletion timestamps, all retained data categories with legal basis per item, pseudonymization actions taken, and audit log reference for every operation. Certificate signed with NoteNova's timestamp authority key.
  6. Confirmation (Day 15): Certificate delivered to institution admin. Erasure request status updated to completed. Audit log entry created for completion.

Special cases: If student had opted into attribution in the Knowledge Reuse vault: their name is stripped from vault entries; content retained as anonymous. Anonymized vault content is not subject to erasure as it is no longer linkable to the data subject. If student is the sole author of a course-critical document (e.g., a wiki entry): document ownership transferred to faculty before personal data deletion; content preserved as institutional content.

EC-06: Network Partition During Real-Time Collaboration ›

Scenario: 6 students are collaborating on a shared document in real time. The network partitions: 3 students lose connectivity while 3 remain connected. Both sides continue editing for 4 minutes before the partition heals.

System Response: The CRDT (Yjs) engine handles this natively by design. Each side maintains its own CRDT state graph. When partition heals: (1) WebSocket reconnects and each client broadcasts its local operation log; (2) CRDT server merges all concurrent operations deterministically — the merge is mathematically guaranteed to produce the same result regardless of merge order; (3) All 6 clients converge to the same document state within 2–3 seconds of reconnection; (4) No data loss on either side of the partition; (5) Version snapshot created automatically at merge point labeled "Auto-merged after network partition [timestamp]"; (6) If any client was offline for >15 minutes: the full document state (not just the diff) is requested on reconnect to handle potential vector clock overflow edge cases.

Pathological case — conflicting block deletions: If both sides deleted different blocks that reference each other (e.g., Side A deleted a block that Side B's new block references via backlink): the orphaned backlink is flagged as a dead reference post-merge and shown to the document author with a "Resolve" prompt.

EC-07: Faculty Account Compromised Mid-Grading ›

Scenario: A faculty member's account credentials are compromised. An attacker accesses the account and begins modifying student grades in the gradebook.

Detection: Anomaly detection (rate of grade changes per minute, login from new geolocation/IP, unusual access time) fires a real-time alert. Security incident created (P1). Admin alerted via PagerDuty and email.

Immediate Response: Admin suspends faculty account and revokes all active sessions within minutes of detection. All grade changes made by the compromised session are flagged in the grade history with a "Security Review Required" tag — not automatically reverted, as legitimate grading may have also occurred in the same session. The attacker session is invalidated and logged with full operation history.

Forensic Review: Admin exports the audit log for the compromised session. Every grade read, write, and export action is timestamped and listed. Session IP and geolocation compared to faculty's normal access patterns. All grade changes from the compromised session isolated for faculty review on account restoration.

Recovery: Faculty account restored with new credentials after identity verification with institution IT. Faculty reviews flagged grade changes: confirms legitimate vs. attacker-made changes. Attacker-made grade changes reverted with audit note. Students whose grades were changed notified (without disclosing security incident details: "Your grade has been reviewed and updated. If you have questions, contact [faculty].").

EC-08: AI-Generated Content in Student Submission Detection Dispute ›

Scenario: Student submits an assignment. AI content detection flags it as 78% likely AI-generated. Student disputes this, claiming they wrote it themselves but used AI tools only for grammar checking (which the assignment policy permits).

System Response: (1) Integrity flag raised with AI detection score and confidence interval (e.g., 78% ± 12%); (2) Flag appears in faculty review queue — not automatically reported to any institutional body; (3) Faculty reviews: the AI detection score is one signal, not a verdict. Faculty examines the submission for additional indicators: writing style consistency, cited sources, draft save history (timestamp pattern showing gradual document growth vs. single large paste event), AI Tutor usage history for this student on this topic; (4) Faculty can: clear the flag (no further action), escalate to department integrity process, or request student explanation via submission clarification chat (M-47); (5) All actions and rationale logged in the integrity event record; (6) If student disputes at institutional level: the full evidence package (AI score, confidence interval, draft history, AI Tutor usage, faculty notes) is exportable for review board.

Design note: The AI detection score is explicitly labeled in all UIs as "probabilistic indicator" not "determination." The grading system prevents faculty from using the AI score alone as grounds for a zero grade — a faculty note with additional evidence is required. This design choice is intentional to prevent over-reliance on imperfect AI detection tools.

Design Flexibility

NoteNova must not impose a single pedagogical model on all institutions. Higher education is not homogeneous: a community college teaching vocational skills, a research university running doctoral programs, and a law school conducting Socratic seminars all have fundamentally different workflows, assessment philosophies, and engagement expectations. The flexibility model described here ensures NoteNova adapts to the institution — not the reverse.

26.1 Configuration vs. Customization Principle

Guiding Principle

Configuration changes behavior through data. Customization changes behavior through code. NoteNova's flexibility model is configuration-first. All institution-specific behavior must be expressible through the tenant configuration schema. Requests for code-level customization per tenant are an architectural failure signal indicating a missing configuration dimension that should be added to the schema — not a trigger for per-tenant code branches.

The practical implication is that when an institution says "we need X to work differently," the first engineering question is always: "Can this be expressed as a configuration value?" Only when the answer is definitively no does the team consider a new feature, which then becomes available to all tenants. Per-tenant code is never the answer.

NoteNova is a multi-tenant, AI-augmented academic knowledge platform designed for higher education institutions. It serves students, faculty, and administrators through a unified system spanning the complete academic lifecycle: content creation, knowledge structuring, course delivery, assignment management, peer collaboration, intelligent assistance, performance analytics, and long-term institutional knowledge retention. 27.1 Product Strategy Summary

NoteNova is built on the premise that higher education's digital infrastructure problem is not a lack of tools — it is a lack of integration, intelligence, and institutional memory. The strategy has three phases:

Phase 3 — Become the Platform

With the Plugin System active and a developer ecosystem growing, NoteNova becomes the platform that institutional vendors build on — attendance systems, ERP connectors, custom assessment tools, research management. NoteNova becomes the academic OS that everything else integrates with.

Modules: M-52 (Plugin System) + ecosystem

Competitive positioning: Network effects. Switching from NoteNova means losing the plugin ecosystem, the knowledge vault, and years of institutional data. Defensible moat.

Purpose

  • Today vs. Upcoming View: Toggle between "today's tasks" and "next 7 days" with day-by-day grouping. Overdue items always pinned at top in red.
  • Completion Animation: Items disappear from center with subtle animation when completed. Daily completion rate shown as progress bar. "All done" state with encouraging message when queue is empty.
  • Action Center Digest: Daily 7 AM email digest of today's Action Center state per user. Configurable time and opt-out. Uses actual personalized data, not generic reminders.
UI Behavior
Action Center opens as the default route post-login. Items grouped by section: "Urgent — do now" (overdue + critical), "Today", "This week". Each item shows: icon (type), title, course name, time remaining or time since. Clicking item navigates to resource and marks item as "in progress." Returning to Action Center shows progress state. Section collapse/expand with count badge. "Snooze until tomorrow" per item (non-critical items only). Real-time update via WebSocket: new items appear without page refresh.
Edge Cases
  • Action Center with 100+ items (end-of-semester): pagination + collapse by course; show total count not individual items beyond 20 per section.
  • Item becomes irrelevant after appearing (e.g., faculty extends deadline): item updates in real time via WebSocket; no stale items.
  • New user with no items: onboarding checklist shown instead of empty state — guides user through first note, first course navigation, and profile setup.
M-47 Contextual Chat Collaboration
Purpose
Provide lightweight, contextually anchored communication threads that eliminate the need for external messaging apps for academic discussions — without becoming a generic chat platform. Every conversation in NoteNova is attached to a specific academic context: an assignment, a note, a course, or a submission. No open-ended free chat rooms that fragment focus.
Features
  • Context Types: Assignment discussion thread (per assignment, all enrolled students + faculty), note discussion (any shared note), course Q&A (course-level; students ask, faculty answers), submission clarification (faculty ↔ student, private), study group chat (per study group, student-only).
  • No Free DMs: Direct messages between students are not supported (prevents misuse, keeps focus academic). Faculty-to-student private messaging is available in submission clarification context only.
  • Thread Structure: Each context has a single main thread. Replies are threaded (2-level depth). Threads are persistent and searchable.
  • Rich Input: Text, inline code, math (LaTeX), file attachment (images, PDFs), link preview. No emoji-only messages (minimum 1 word of text).
  • AI Thread Summary: For long threads (>20 messages), AI generates a rolling summary pinned at top: "Key points so far: ..." Updates every 10 new messages.
  • Upvote Questions: In course Q&A, students can upvote a question. Faculty sees questions sorted by upvotes. Answered questions marked with green checkmark by faculty. Promotes efficient prioritization of common doubts.
  • Read Receipts for Faculty: Faculty can see how many students have read a message (count only, not names) in course Q&A. In submission clarification, full read receipt with timestamp.
  • Search: Full-text search within a context thread and across all threads user has access to. Filter by person, date, attachment type.
  • Notifications: @mention triggers notification. Faculty reply to student question triggers notification. New message in active study group thread triggers notification (rate-limited).
Edge Cases
  • Student attempts to use chat to request answers to graded assignment: AI content policy flag; message not blocked but logged; faculty notified if pattern detected.
  • Faculty inactive in course Q&A for >72 hours with unanswered questions: admin notified; optional escalation to department coordinator.
  • Chat in offline mode: messages queued locally; posted on reconnect with original timestamp preserved; other participants see message in order of server receipt.
  • Thread exceeds 500 messages: archive older messages with "Load earlier" pagination; AI summary always reflects full thread history including archived messages.

M-48 File Distribution System Content Delivery
Purpose
Replace the dependency on Google Drive, Dropbox, or email attachments for distributing course files by providing an integrated, structured, versioned file sharing system within NoteNova that auto-organizes files by topic, supports instant access for enrolled students, and maintains a clean course resource library across semesters.
Features
  • Course Resource Folder: Each course has a structured file library. Top-level folders are auto-created per course module. Faculty drag-and-drop files into the appropriate module folder. Students access the same folder structure in read-only mode.
  • Instant Distribution: Faculty uploads a file → students with active sessions see it appear in the file library within 5 seconds via WebSocket push. No page refresh required.
  • File Versioning: Faculty can replace a file (e.g., updated slide deck) without breaking existing links. Previous versions accessible via version history dropdown. Students who download the old version are shown a banner on next visit: "This file has been updated."
  • AI Auto-Tagging of Files: On upload, AI analyzes file content (OCR for PDFs, text extraction for docs) and suggests topic tags and course module mapping. Faculty approves in one click.
  • Bulk Upload: Drag a folder or ZIP; auto-extracted and organized. Duplicate detection (hash-based): duplicate files shown with "already exists" notice before upload completes.
  • Access Control: Files inherit course enrollment permissions by default. Faculty can restrict individual files to specific student groups or make them time-gated (available from a specific date).
  • Download Analytics: Faculty can see download count per file and which students have not downloaded key required materials (configurable "required" flag per file).
  • Storage Quota per Course: Admin-defined per tenant (default 10 GB per course). Visual usage indicator for faculty. Warning at 80%, block at 100%.
  • Semester Carry-over: Faculty can select files to carry over to next semester's course instance. Files are copied, not moved (preserving history in original course).
Edge Cases
  • File contains sensitive student data uploaded by mistake: faculty can delete within 24 hours; system checks file name patterns for PII indicators on upload and warns before completion.
  • Same file uploaded by multiple faculty across different courses: files are separate instances; no cross-course file deduplication at storage layer (isolation maintained).
  • Student downloads file while offline: pre-cache for offline if student has used "Make available offline"; otherwise, download queued and initiated automatically on reconnect.
M-49 Submission Timeline & Audit View Assessment
Purpose
Give faculty a complete, chronological view of the submission state for every student across every assignment — providing instant visibility into who has submitted, who is late, who has not started, and the full submission history per student including attempt timestamps, file versions, and integrity check results. This is the operational view faculty need to manage a course without opening each submission individually.
Features
  • Assignment-Level Timeline: For a selected assignment: visual timeline showing submission events across the submission window. Dots on the timeline represent each student submission (color-coded: green = on time, amber = late, red = missing, grey = extension granted). Hover a dot to see student name, exact timestamp, and attempt number.
  • Student-Level Detail: Click any student to expand: full attempt history (timestamp per attempt, file metadata per version, integrity check result per attempt, draft-save timestamps showing work in progress). Faculty can see if a student opened the assignment but never submitted.
  • Status Summary Cards: At-a-glance counts: Submitted on time / Submitted late / Missing / Extension granted / Integrity flagged / Grading complete / Grading pending.
  • Bulk Actions from Timeline: Select students by status; send targeted reminder; grant bulk extension; mark missing as zero grade; download selected submissions as ZIP.
  • Late Submission Breakdown: For late submissions: shows how late (hours/days), whether late penalty was applied, faculty override note if penalty waived.
  • Cross-Assignment View: Faculty toggle to see a student's submission history across all assignments in the course — identifying patterns (always late, always last-minute, first to submit).
  • Export: Full submission log exportable as CSV with all fields (student ID, submission timestamp, attempt number, file hash, integrity score, grade). Suitable for academic integrity proceedings.
UI Behavior
Default view is timeline for most recent open assignment. Switching assignments via dropdown re-renders timeline without page reload. Color legend always visible in top-right. Clicking a student dot opens a side drawer with full detail (does not navigate away from timeline). Side drawer has "Open full submission" button for deep-link to grading view. Edge Cases

  • Student submission count very high (1,000+ students): timeline dots cluster; zoom in to specific time range; progressive load on zoom.
  • Submission timestamp dispute (student claims they submitted on time): audit trail shows exact server-received timestamp vs. student's claimed time; system clock used as authoritative source with NTP synchronization noted in export.
M-50 AI Study Planner AI / Engagement
Purpose
Generate personalized, adaptive daily study plans for students that are grounded in actual course deadlines, individual performance data, identified knowledge gaps, and cognitive load principles — making NoteNova the first thing a student consults each morning to know what to study and why. This is the primary daily habit hook that differentiates NoteNova from every passive LMS.
Features
  • Plan Generation: AI generates a recommended daily study plan based on: (a) deadlines in the next 7 days (from Calendar module), (b) topics with low quiz/assessment performance (from Analytics module), (c) flashcard review due count (from Smart Flashcards module), (d) course materials not yet accessed, (e) user-defined study time availability per day (set in preferences).
  • Study Block Format: Plan is presented as ordered time blocks: "Block 1 — 30 min — Review Week 4 Lecture Notes (Thermodynamics: topic you scored 42% on in Quiz 2)." Each block has a direct link to the relevant resource.
  • Workload Balancing: AI distributes study load evenly across available days before a deadline rather than recommending everything the night before. Applies a spaced practice model: topics reviewed multiple times in decreasing intervals before exam.
  • Adaptive Re-planning: If a student completes a block and marks it done, the plan re-calculates remaining blocks. If a new assignment is published with a close deadline, plan updates automatically and notifies student of change.
  • Weak Topic Focus: Plan weights study time toward topics where the student has demonstrated low performance (quiz scores, incorrect flashcard frequency, AI tutor question history on that topic).
  • Plan Acceptance and Override: Student can accept plan as-is, remove individual blocks, swap order, or regenerate with different priorities. Overrides are respected; AI does not re-add removed blocks for 48 hours.
  • Weekly Review Summary: Every Sunday, AI generates a "This week in review" summary: topics studied, flashcards reviewed, assessments completed, performance vs. last week. Delivered as in-app card and optional email.
  • Goal Integration: Student sets a target grade per course. AI factors this into planning intensity recommendations: "To reach your 85% target in Economics, you need to improve performance in monetary policy topics."
Edge Cases
  • Student has zero performance data (new to platform): plan based only on deadline proximity; AI states "plan will improve as you complete more assessments."
  • Student marks every block done without actually studying (gaming): plan still generated; AI does not police completion; faculty analytics are separate and based on actual submission/quiz performance.
  • All deadlines are >14 days away (no urgency): plan focuses on flashcard reviews and pre-reading; shows "low-urgency week" label clearly.
  • Student available study time input is 0 hours (exam week setting): plan generates minimum viable actions only (e.g., review AI summary of key topics: 10 minutes).
Data Sources
Calendar module (deadlines), Analytics module (performance scores), Smart Flashcards module (due cards and retention rate), AI Tutor module (question history by topic), Content Service (accessed vs. not-accessed materials), Assignment Engine (upcoming assignments). No external data used.
M-51 Knowledge Reuse Layer Institutional Memory
Purpose
Solve the most structurally damaging problem in higher education identified in Section 2.3: knowledge loss between cohorts. The Knowledge Reuse Layer provides controlled, faculty-governed access to structured knowledge artifacts from previous academic terms — anonymized student notes, exemplary submissions, best answers, past assignment evolution, and faculty-curated legacy resources — making each new cohort a beneficiary of all prior institutional learning rather than starting from zero.
Features
  • Legacy Knowledge Vault: At course archive, faculty can select content to promote to the Legacy Vault: exemplary student submissions (anonymized by default), faculty notes, lecture recordings, annotated readings, high-scoring assignment responses. Content is tagged by course, topic, and academic year before vaulting.
  • Cohort Access Control: Faculty decides per vault item: (a) accessible to current students only, (b) accessible to all future cohorts automatically, (c) faculty-only (for curriculum planning). Anonymization is enforced by platform; student identity stripped before vaulting unless student explicitly opts in to attribution.
  • Discovery Interface: Students see a "From Previous Cohorts" section in each course module (if faculty has enabled it). Content is labeled with academic year and type (e.g., "Exemplary Essay — 2024 cohort"). Searchable by topic tag.
  • Faculty Curation Dashboard: Faculty reviews what is in the vault for their course. Can remove any item. Can add curator notes ("This essay excels at X but misses Y — study it critically"). Can compare how best answers have evolved across years.
  • AI Synthesis Across Cohorts: AI can generate a "what previous students found most challenging in this topic" insight from vault content — giving current students and faculty a head start on common difficulty areas. Generated as read-only advisory; not shown as authoritative.
  • Opt-In Attribution: Student completing a course can opt in to having their best work attributed to them in the vault (with name visible to future cohorts). Requires: student explicit consent, faculty nomination, admin audit record. Creates a legacy academic portfolio element.
  • Department Vault: Department admins can promote cross-course vault items to a department-level knowledge base: foundational concept explanations, cross-disciplinary resources, accreditation evidence.
Privacy & Compliance
Default behavior is full anonymization before vaulting (student name, ID, email stripped from document and metadata). Anonymization runs as a server-side preprocessing step; no manual faculty action required. FERPA review: vaulted content is academic work product, not an education record in the FERPA-restricted sense once anonymized, but institution's legal team should confirm per jurisdiction. Vault content is subject to tenant data retention policy and can be purged on request.
Edge Cases
  • Student whose work was vaulted (anonymized) later requests GDPR erasure: anonymized content has no re-linkable identifier; erasure cannot re-identify original author. If student had opted into attribution, their name is removed and content retained as anonymous per default policy.
  • Vaulted "exemplary" submission later discovered to be plagiarized: faculty removes from vault immediately; audit trail records removal reason; current students who accessed it are notified it has been retracted.
  • Faculty attempts to vault content from a different course they do not teach: blocked; vault access is strictly course-scoped for faculty.
M-52 Plugin & Extension System Platform Extensibility
Purpose
Enable institutions and third-party developers to extend NoteNova's capabilities through a governed plugin architecture — without requiring changes to the NoteNova core codebase. This transforms NoteNova from a closed platform into an institutional academic operating system that universities can adapt to their specific ecosystem: attendance systems, ERP integrations, custom assessment tools, departmental workflows, and institution-specific compliance modules.
Architecture
  • Plugin Types: (a) UI Extension — adds a panel, tab, or widget to existing NoteNova screens; (b) Data Connector — reads from or writes to NoteNova data via a governed API scope; (c) Workflow Hook — triggers on NoteNova events (submission created, grade released, user enrolled) and performs external actions; (d) AI Extension — registers a custom prompt or retrieval source in the AI Orchestration pipeline (tenant-approved only).
  • Plugin Manifest: Each plugin declares: name, version, author, required API scopes, event subscriptions, UI injection points, data access requirements, and privacy impact rating. Admin reviews manifest before installation.
  • Sandbox Execution: UI extensions run in a sandboxed iframe with a limited postMessage API; cannot access parent DOM or cookies directly. Data connector plugins run server-side in isolated Lambda-style compute with network egress restricted to declared allowlisted domains.
  • Marketplace / Registry: Institution-private plugin registry (admin-managed) plus optional access to NoteNova-verified public plugin registry (Anthropic-reviewed). Public plugins have security audit badge and data handling certification.
  • Permission Model: Plugins request specific OAuth-style scopes: courses:read, submissions:write, grades:read, users:read. Scopes must be approved by Super Admin. Users see a permission summary before a plugin activates for their account.
  • Example Plugins: Attendance tracker (marks attendance per lecture session, syncs with course analytics), ERP integration (reads student enrollment from university ERP, writes grades back), custom plagiarism provider (replaces built-in with institution-licensed tool), accessibility plugin (adds screen-specific custom tools), research ethics approval tracker (links submissions to ethics board record).
  • Plugin Isolation: A malfunctioning plugin cannot crash or corrupt core NoteNova data. Plugin errors are caught, logged, and reported to plugin author; core UI continues functioning. Automatic plugin disable on error rate >5% over 1 hour.
Developer Experience
Plugin SDK available (TypeScript). Local development server with NoteNova mock environment. Plugin CLI for scaffold, test, build, and publish. Webhook event simulator for testing workflow hook plugins. Full API reference with plugin-specific endpoints. Plugin developer portal with usage analytics (how many tenants installed, active users, error rate).
Edge Cases
  • Plugin requests scope it does not need (over-privileged): flagged during manifest review; admin can install with reduced scope if plugin functionality allows.
  • Plugin author discontinues support: plugin enters "deprecated" state; admin warned; plugin disabled automatically after 90-day grace period unless admin explicitly extends.
  • Plugin attempts to exfiltrate data to non-declared domain: network egress blocked at infrastructure level; security incident logged; plugin auto-disabled; author account suspended from registry.
Governance
All plugins installed in a tenant are logged in the Audit Log (M-32). Plugin data access is subject to the same tenant data isolation guarantees as core platform. Plugins cannot read data from tenants they are not installed in. NoteNova retains the right to revoke any public plugin from the registry for policy violations.
